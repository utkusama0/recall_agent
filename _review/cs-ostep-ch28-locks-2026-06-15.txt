A synchronization primitive that grants exclusive access to a critical section — at most one thread may hold it at any time.<br>While held, all other threads attempting to acquire it are blocked or spin. TAS atomically reads a memory word, writes 1 to it, and returns the original value — all as a single indivisible operation.<br>Atomicity prevents two threads from both reading 0 and both believing they acquired the lock. A spinning thread consumes its entire scheduling quantum doing no useful work.<br>On a uniprocessor this also prevents the lock-holder from running, potentially causing deadlock if preemption is disabled. Blocking requires a syscall: the kernel deschedules the thread, places it in a wait queue, and later reschedules it.<br>This context switch overhead is typically hundreds to thousands of nanoseconds — far more than a single TAS instruction. Spin lock: busy-waits (polls in a loop), very low overhead, wastes CPU while waiting.<br>Sleep lock: blocks via syscall, higher per-operation cost, but yields the CPU.<br>Choice: prefer spin when expected wait time &lt; context-switch cost. When the expected wait time (≈ critical section duration) is shorter than the cost of one context switch.<br>If waiting is brief, the context-switch overhead of a sleep lock exceeds the total cost of spinning. The spinning thread monopolizes the only CPU, so the thread holding the lock is never scheduled and never releases it.<br>The spinner waits forever: a true deadlock, not livelock. cs-ostep-ch28-locks-010	What is the mutex lock invariant?	At most one thread holds the lock at any instant while the lock is in the acquired state.<br>Equivalently: the lock word equals 1 if and only if exactly one thread has successfully acquired it. The read and write of the lock word must be indivisible — no other thread can observe or modify the lock word between them.<br>If TAS were non-atomic, two threads could both read 0 before either writes 1, allowing double acquisition.
